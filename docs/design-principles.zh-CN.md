# 设计原则

> 核心主张：技能必须是可验证的。  
> 当你改 `SKILL.md` 时，先改 `tests/`，并确保 `npm run eval` 能通过。

## 1. 失败驱动演进
只围绕失败模式建规约，不凭感觉编规则。每条约束都应该对应至少一个真实复现路径。

## 2. 证据优先
对任何能力声明，都需要可复现证据。  
- 代码必须实测通过
- 性能声明需要数据
- 架构选择需要权衡记录

## 3. 最小惊讶原则
- 读完 `SKILL.md` 之后，用户不应该感到困惑
- 禁止事项要明确
- 期望行为要可操作

## 4. 对称性约束
当规约施加了义务（例如“必须检查”），它必须同时说明：
- 检查什么
- 检查不到该怎么办
- 什么时候可以跳过

## 5. 负载与范围
不要让技能膨胀成“什么都管”。  
每项规约都应该：
- 解决一个明确的失败模式
- 在 `SKILL.md` 中有精确措辞
- 在 `tests/` 中有可验证案例

## 6. 行为可测试
规则的效果应该能通过 `npm run eval` 评估。如果不能，说明规则太模糊或评测集太弱。
